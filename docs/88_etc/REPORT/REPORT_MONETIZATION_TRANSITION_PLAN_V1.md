## 2. Type B vs Type C 핵심 비교

### 2-1. 사용자 관점

| 항목 | Type B | Type C |
|------|--------|--------|
| **진입 장벽** | 높음 — API 발급, 카드 등록 (해외 결제 포함) 필요 | 낮음 — 국내 카드로 바로 충전 |
| **비용 예측** | 월 고정 → 예측 가능하나 API 별도 | 쓴 만큼 → 사용량 조절 가능 |
| **API 비용 체감** | 직접 부담 → 사용량 많을수록 불만 가능 | 없음 → 단가가 높아도 체감 낮음 |
| **심리적 부담** | 구독 약정 부담 (매월 결제 부담감) | 소액 충전 → 부담 낮음 |
| **활발한 사용 유도** | 구독 유지 위해 많이 써야 본전 | 소진 후 재충전 → 필요 시만 사용 |

### 2-2. 운영자(우리) 관점

| 항목 | Type B | Type C |
|------|--------|--------|
| **수익 예측성** | 높음 — MRR(월 반복 매출) 구조 | 낮음 — 충전 타이밍 불규칙 |
| **마진율** | 매출 100% = 순이익 (API 비용 없음) | 88~92% (API 비용 ~8~12% 차감) |
| **API 비용 리스크** | 없음 (사용자 부담) | 있음 — API 가격 인상 시 마진 감소 |
| **운영 복잡도** | 낮음 | 높음 — 크레딧 원장, API 키 풀 관리 필요 |
| **고객 이탈 시** | 다음 달 자동 취소 → 수익 단절 | 잔여 크레딧 환불 이슈 발생 가능 |
| **결제 실패 처리** | 재결제 시도 로직 필요 | 선불 → 결제 실패 없음 |

### 2-3. 개발 복잡도 비교

| 기능 | Type B 필요 | Type C 추가 필요 |
|------|------------|----------------|
| 회원가입/로그인 (JWT) | ✅ | ✅ (공통) |
| 구독 결제 (Toss Payments) | ✅ | - |
| 크레딧 충전 결제 | - | ✅ |
| API 키 입력/암호화 저장 | ✅ | - (우리 키 사용) |
| 크레딧 원장 (잔액/차감/이력) | - | ✅ |
| API 키 풀 관리 (운영자) | - | ✅ |
| 파이프라인 트리거 | ✅ | ✅ (공통) |
| 발행 이력 대시보드 | ✅ | ✅ (공통) |
| 구독 갱신/취소/웹훅 | ✅ | - |
| 크레딧 만료 알림 | - | ✅ |
| 소비자보호법 환불 처리 | - | ✅ (크레딧 정책) |

**결론**: Type B가 Type C보다 **개발 복잡도 약 40% 낮음**.

## 3. 구현 순서 권장 및 근거

### ✅ 권장: Type B 먼저, Type C 추가

```
Phase 24 → [Free + Type B] MVP 출시
Phase 25 → [Type C] 크레딧 모델 추가
```

### 근거

| 이유 | 설명 |
|------|------|
| **빠른 수익화** | Type B가 단순 → 더 빨리 출시 → 더 빨리 매출 발생 |
| **API 비용 리스크 0** | BYOK이므로 구독자가 늘어도 운영자 API 비용 증가 없음 |
| **MRR 구조 확보** | 구독은 매월 반복 수익 → 사업 안정성 |
| **학습 후 Type C 설계** | B 운영하며 실제 사용 패턴 파악 → C 단가/크레딧 설계에 반영 |
| **Type C는 B 위에 얹기** | 공통 인프라(auth, pipeline, dashboard) 재활용 → 추가 개발량 최소화 |

### ❌ Type C 먼저 하면 안 되는 이유

1. API 비용 관리 복잡도가 높아 초기 오버엔지니어링
2. 크레딧 원장 버그 발생 시 즉각 사용자 피해 (환불 분쟁)
3. 사용자가 얼마나 쓸지 모르는 상태에서 단가 설정이 불확실
4. MRR 없이 충전 방식만으로는 수익 예측 불가능

```
✅ 사용자가 이메일로 회원가입할 수 있다

✅ 구독이 매월 자동 갱신되고 취소할 수 있다
✅ Free 사용자는 월 20개 포스팅 제한이 적용된다

```

## 6. 주요 고려사항

### 6-1. 법적/컴플라이언스

| 항목 | 내용 | 적용 시점 |
|------|------|---------|
| **투자 권유 금지** | 서비스 내 면책사항 명시 필수 ("본 콘텐츠는 투자 권유가 아님") | Phase 24 |
| **전자상거래법** | 청약 철회 14일 보장 (디지털 콘텐츠 예외 조항 확인 필요) | Phase 24 |
| **개인정보보호법** | 회원 정보 수집/이용/파기 정책, 개인정보처리방침 페이지 | Phase 24 |
| **크레딧 유효기간** | 충전일로부터 2년 권장, 만료 30일 전 안내 의무 | Phase 25 |
| **API 키 보안** | 사용자 API 키 AES-256 암호화, DB에 평문 저장 금지 | Phase 24 |
| **금융소비자보호법** | 투자정보 서비스 해당 여부 검토 (정보 제공 vs 투자 자문) | Phase 24 이전 |

> ⚠️ **금융소비자보호법 주의**: "투자 분석 글 자동 작성"이 투자 자문업으로 분류될 경우 별도 등록 필요. 현재는 "정보 제공"으로 유지 + 강력한 면책사항으로 커버.

### 6-2. 기술적 고려사항

#### WordPress 연동 방식 결정 (중요)

세 가지 옵션 중 선택 필요:

| 옵션 | 방식 | 장점 | 단점 |
|------|------|------|------|
| **A: BYOWP** (권장) | 사용자가 자신의 WP에 연결 | 인프라 비용 없음, 확장성 무제한 | 사용자 WP 설치 필요 |
| B: 공유 WP | macromalt 단일 WP에 사용자별 카테고리 | 온보딩 쉬움 | WP 서버 비용 발생, 콘텐츠 혼재 |
| C: WP 호스팅 제공 | 사용자별 WP 인스턴스 | 최고 UX | 운영 복잡도 폭발 |

**권장: Option A (BYOWP)** — 사용자가 자신의 WP URL + 앱 비밀번호를 입력하는 방식. 현재 코드 구조와 동일.

#### 파이프라인 동시 실행 관리

```
동시 실행 유저 10명 → 각 파이프라인 45분 소요
→ 서버 큐 없으면 동시 요청 충돌 가능

해결: 작업 큐 (Celery + Redis or RQ) 도입
- Phase 24 MVP: 동시 실행 사용자 5명 제한 (소프트 캡)
- Phase 26: 큐 시스템 정식 도입
```

#### API 키 암호화

```python
# AES-256 암호화 저장 (절대 평문 DB 저장 금지)
from cryptography.fernet import Fernet
ENCRYPT_KEY = os.getenv("API_KEY_ENCRYPT_SECRET")  # 별도 환경변수
cipher = Fernet(ENCRYPT_KEY)
encrypted_key = cipher.encrypt(user_api_key.encode()).decode()
# DB에 encrypted_key 저장
```
